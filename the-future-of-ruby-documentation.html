What RDoc does

  • Ruby's default documentation tool — ships with Ruby since 1.8
  • Parses Ruby source, C extensions, and markup files
  • Generates HTML docs and ri terminal docs
  • Supported markup: RDoc, Markdown, RD, TomDoc
  • Powers docs.ruby-lang.org
04

Where we were

RDoc has served Ruby well for 20+ years. But the ecosystem evolved — and so did expectations.

  • Developers now expect dark mode, mobile support, live preview
  • Markdown became the universal documentation format
  • Ruby syntax grew beyond what the parser was built for
  • Type systems emerged — documentation didn't reflect them

This past year, we focused on closing that gap.

Why Markdown?

  • Markdown is the industry standard
  • RDoc markup is Ruby-specific — not widely adopted even in the community
  • Lowers the barrier for contributing to Ruby core docs

If you can write a GitHub README, you can write Ruby documentation.

Can RBS replace :call-seq:?

:call-seq:

# :call-seq: # readlines(sep=$/) -> array # readlines(limit) -> array # readlines(sep, limit) -> array

RBS #:

#: (?String sep) -> Array[String] #: (Integer limit) -> Array[String] #: (String, Integer) -> Array[String]

The overlap is significant. For a language that doesn't want typing, writing type signatures for documentation is huge.

24

The gap: default values

:call-seq:

commercial(cwyear, cweek=1, cwday=1, sg=nil)

Shows actual default values

RBS:

(Integer, ?Integer, ?Integer, ?Symbol?) -> Date

Only says "optional" — default is lost

If we want to migrate, RBS needs default value support.

25

What's next for RBS in docs

  • Aiming to ship on docs.ruby-lang.org
  • Migration path from call-seq: TBD — depends on RBS default value support
  • Community discussion needed

Long-term: structured, machine-readable signatures replace free-form text

Getting ahead

llms.txt: we evaluated it

We prototyped it. Then we looked at the data:

  • 10% adoption across 300K domains
  • Zero measurable impact on AI citations
  • ML model improved when llms.txt was removed
  • Zero bot visits from GPTBot, ClaudeBot, etc.
  • No major LLM provider officially supports it

Trade-off: generation time + code complexity for negligible benefit → not pursuing

28

Better foundations

Instead of special files for LLMs:

  • Markdown — universally understood by humans AND machines
  • RBS types — structured, machine-readable signatures
  • Clean architecture — enables tooling we can't predict
