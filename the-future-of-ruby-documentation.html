RDoc hasn't had major investment in years. That changed this past year.

What RDoc does

  • Ruby's default documentation tool — ships with Ruby since 1.8
  • Parses Ruby source, C extensions, and markup files
  • Generates HTML docs and ri terminal docs
  • Supported markup: RDoc, Markdown, RD, TomDoc
  • Powers docs.ruby-lang.org
04
Quick primer for anyone who hasn't thought about RDoc directly. Most people have seen RDoc output — docs.ruby-lang.org is generated by it — but don't think about the tool itself. It ships with Ruby, handles multiple markup formats, and powers both the HTML docs and the ri terminal docs.

Some context

RDoc has been Ruby's documentation tool for 20+ years. It works. But a lot has changed since it was written.

  • Developers expect dark mode, mobile support, live preview
  • Markdown became the standard way to write documentation
  • Ruby syntax grew beyond what the parser was built for
  • Type annotations exist now — documentation doesn't show them
05
Be factual about where things stood. This isn't about blaming anyone — the ecosystem changed. Developers now expect things that didn't exist when RDoc was written. Dark mode, mobile layouts, live preview — these are table stakes now, and RDoc didn't have them.

Getting there

This is the meatiest section — active work that isn't finished yet. Two big topics: Markdown support and RBS type signatures.

Why Markdown?

  • Markdown is the industry standard
  • RDoc markup is Ruby-specific — even within the community, it's not widely known
  • Many contributors already write Markdown in RDoc comments without realizing it doesn't render correctly

If you can write a GitHub README, you can contribute to Ruby documentation.

15
The case for Markdown is straightforward. It's the industry standard. RDoc markup is Ruby-specific — even within the Ruby community, it's not widely known. I've seen contributors write Markdown syntax in RDoc comments without realizing it won't render.

The URL for standard library documentation already changed in Ruby 3.4: standard_library_rdoc.html became standard_library_md.html. The direction is set.

Markdown was added ~2012 by reusing RDoc's pipeline. Every Markdown fix is a two-format fix.

16
This is why Markdown support took so long. Both parsers — Markdown and RDoc — feed into the same shared pipeline. The Markdown parser outputs RDoc markup strings, which then get re-parsed by the shared InlineParser and formatters.

This means every Markdown fix is also a two-format fix. You can't change the shared code without verifying both formats still work.

Example: ~~strikethrough~~

# Markdown parser outputs: ~~text~~ → "<del>text</del>" # plain string! # Shared InlineParser receives that string # ...but didn't recognize <del> # Strikethrough was silently broken for years # Fix: modify the shared InlineParser # → must verify RDoc markup still works too

Every Markdown fix is a two-format fix.

17
One concrete example. Markdown strikethrough — ~~text~~ — the parser outputs it as a <del> HTML string. That string feeds into the shared InlineParser, which didn't recognize <del>. So strikethrough was silently broken for years.

To fix it, you modify the shared InlineParser — but then you have to verify RDoc markup still works too. That's why progress on Markdown has been incremental.

What's improved

  • ~~Strikethrough~~ aligned with GFM spec
  • GitHub-style heading anchors (#heading-name)
  • Table parsing: inline markdown, incomplete rows
  • Bash/shell syntax highlighting
  • Backtick quoting in RDoc markup too
  • break_on_newline enabled by default

NEW GFM spec comparison test suite

18
Walk through the improvements. Strikethrough aligned with the GFM spec. GitHub-style heading anchors. Table parsing handles inline markdown and incomplete rows now. Bash syntax highlighting. Backtick quoting works in RDoc markup too.

The big one at the bottom: we now have a GFM spec comparison test suite that systematically tracks our compatibility. Instead of fixing bugs one-by-one, we can see exactly where we stand.

RDoc markup vs Markdown

RDoc markup

= Heading *bold* _italic_ +code+ {link text}[url] * bullet item

Markdown

# Heading **bold** *italic* `code` [link text](url) - bullet item
19
Same documentation in both formats. The Markdown version on the right is immediately recognizable to any developer. The RDoc markup on the left — most people have to look up the syntax.

Migration plan

1Get Markdown to GFM-level quality ← in progress
2Build migration tooling (RDoc markup → Markdown)
3Migrate ruby-core documentation
4Deprecate RDoc markup? (open question)

The URL already changed: standard_library_rdoc.html → standard_library_md.html in Ruby 3.4+

20
Multi-step process. Step 1 is where we are now — getting Markdown to GFM-level quality. Steps 2 through 4 are future work. Step 4 is an open question — I'm not sure deprecating RDoc markup is the right move, and that's a community decision, not just mine.

Note: the URL already changed in Ruby 3.4. That's a real signal about direction.

RBS type signatures in documentation

# Inline RBS annotation #: (String name, ?Integer age) -> User def create_user(name, age = nil) # ... end

RDoc extracts #: annotations and displays type signatures in HTML and ri. Type names are linked to their documentation.

21
RBS inline annotations use the #: syntax. You write the type signature directly above the method. RDoc parses these, validates them through the RBS parser, and displays them in both HTML output and ri terminal docs.

The type names in the output are linked — clicking String takes you to String's documentation page.

Same information in the terminal. When you run ri, the type signatures appear right below the method name. Same structured data, but formatted for the terminal instead of a browser.

Can RBS replace :call-seq:?

:call-seq:

# :call-seq: # readlines(sep=$/) -> array # readlines(limit) -> array # readlines(sep, limit) -> array

RBS #:

#: (?String sep) -> Array[String] #: (Integer limit) -> Array[String] #: (String, Integer) -> Array[String]

The overlap is significant. For a language that doesn't want typing, writing type signatures for documentation is a big shift. But I think it's the right direction — structured signatures are better for both humans and tools.

24
I want to raise this as a question, not a declaration. Look at the overlap — both express the method name, parameters, return type, and overloads. The formats are different, but the information is the same.

I think structured types are the right direction for documentation. For a language that has explicitly chosen not to require typing, the fact that people are already writing type signatures for documentation purposes — that's significant. It tells you something about what documentation actually needs.

But this is a community decision, not just mine. I have an opinion, and I'll share it, but the migration path needs broader input.

The gap: default values

:call-seq:

commercial(cwyear, cweek=1, cwday=1, sg=nil)

Shows actual default values

RBS:

(Integer, ?Integer, ?Integer, ?Symbol?) -> Date

Only says "optional" — default is lost

If we want to migrate, RBS needs default value support.

25
Here's the concrete blocker. call-seq shows actual default values — cweek=1, cwday=1. RBS can only say the parameter is optional — it can't express what the default is. This matters especially for C extensions, where there's no Ruby source to inspect.

If we want to migrate from call-seq to RBS, RBS needs to support default values. That's not solved yet.

Where this stands

  • I'm aiming to ship RBS display on docs.ruby-lang.org
  • The call-seq migration depends on RBS supporting default values — that's not solved yet
  • This needs community discussion, not just my opinion

But the direction seems clear: structured, machine-readable signatures are better than free-form text.

26
Status update. I'm aiming to ship RBS display on docs.ruby-lang.org. The call-seq migration is a longer conversation that depends on RBS getting default value support and on community input.

But the direction seems clear to me: structured, machine-readable signatures are better than free-form text — for humans reading docs and for tools consuming them.
03

Getting ahead

So we've caught up on the basics, and we're making real progress on Markdown and types. The question I kept getting asked was: what about AI? What about LLMs?

So I looked into it.

llms.txt: I prototyped it

Then I looked at the data:

  • 10% adoption across 300K domains
  • Zero measurable impact on AI citations
  • ML model improved when llms.txt was removed
  • Zero bot visits from GPTBot, ClaudeBot, etc.
  • No major LLM provider officially supports it

The trade-off is generation time and code complexity for every RDoc user, for a benefit that doesn't exist yet. So I decided not to pursue it — at least for now.

28
I actually prototyped llms.txt generation for RDoc. Then I looked at the data before committing to it.

SE Ranking studied 300,000 domains — only 10% adoption. An XGBoost model for predicting AI citations actually improved when llms.txt was removed as a variable. Zero bot visits from GPTBot, ClaudeBot, or PerplexityBot to the docs. No major LLM provider officially supports it.

The trade-off is real: generation time and code complexity for every RDoc user, for a benefit that doesn't measurably exist. So I decided not to pursue it — at least for now. If the data changes, I'll revisit.

What I think actually helps

Instead of generating special files for LLMs, I'd rather make the documentation itself better:

  • Markdown — any tool can parse it, humans and machines alike
  • RBS types — structured data instead of free-form text
  • Clean architecture — makes it easier to build things we haven't thought of yet

On the tooling side, I've been building github.com/st0012/ruby-skills — Claude Code plugins for Ruby development. No RDoc-specific skills yet, but that's coming once the Markdown work is further along.

29
Instead of chasing a trend, I'd rather build foundations that naturally serve all consumers — humans and machines. Markdown is universally parseable. RBS types are structured data. Clean architecture makes it easier to build things we haven't predicted yet.

On the tooling side, I've been building ruby-skills — Claude Code plugins that teach AI assistants about Ruby development. 109 stars, actively used. No RDoc-specific skills yet, but that's coming once the Markdown work is further along.

How I think about this

  • Check the data before building a feature — llms.txt looked promising until I looked at the numbers
  • Invest in foundations that serve multiple purposes — Markdown, types, and clean architecture all make docs better for everyone, including future tools
  • As a maintainer, I'd rather ship fewer things that are solid than chase every trend

Tying it together. Check the data before building — llms.txt looked promising until I checked the numbers. Invest in foundations that serve multiple purposes. As a maintainer, I'd rather ship fewer things that are solid than chase every trend.
