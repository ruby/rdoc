# Type Signatures & AI Performance — Deep Research

## Source 1: Mundler et al., PLDI 2025

**Paper:** "Type-Constrained Code Generation with Language Models"
**Authors:** Niels Mundler, Jingxuan He (ETH Zurich), Hao Wang, Koushik Sen, Dawn Song (UC Berkeley), Martin Vechev (ETH Zurich)
**URL:** arxiv.org/abs/2504.09246

### Methodology
- **Language:** TypeScript only
- **Models:** 6 open-weight LLMs (2B-34B): Gemma 2 (2B/9B/27B), DeepSeek Coder 33B, CodeLlama 34B, Qwen2.5 32B
- **Benchmarks:** HumanEval (159 instances), MBPP (384 instances)
- **Tasks:** Synthesis, translation (Python→TS), repair
- **NOT tested:** GPT-4, Claude, or any commercial API model (approach requires next-token probability access)

### The "94%" Claim
Page 171:2, exact text: *"on average 94% of compilation errors result from failing type checks"*

Context: This is from their own evaluation in Section 5. Complementary stat (p.171:4): *"only 3.5% of the functional errors and 6% of the compilation errors in LLM-generated code are due to syntactic errors."*

**What it actually means:** In TypeScript code generated by 2B-34B open-weight models on HumanEval/MBPP, 94% of compilation failures are type errors, not syntax errors.

**What it does NOT mean:** "94% of all AI code errors are type errors." Logic errors, runtime errors, wrong algorithms — none captured.

### Key Results

**Compilation error reduction (Table 2):**
Type-constrained decoding reduces compiler errors by 74.8% (HumanEval) and 56.0% (MBPP), compared to only 9.0% and 4.8% from syntax-only constraining.

**Functional correctness pass@1 improvement (Table 3):**
- Synthesis: +3.5% (modest)
- Translation: +5.0% (modest)
- Repair: +37.0% (substantial — most relevant for AI coding assistants)

**For 99.4% of cases**, only one loop iteration needed — the LLM's first-choice token is already type-valid.

### Common Type Errors (Figure 9)
- Wrong parameter count (Python allows `split()`, TS requires argument)
- Missing return statement (returns `undefined` violating type)
- Incorrect type inference (`reduce` with `[]` infers `never[]`)

### Limitations (from the authors)
1. Building a completion engine per language "currently requires manual efforts"
2. Even strong LLMs struggle on complex tasks: 18% compilation error rate for o1-mini, 39% for DeepSeek R1, 27% for Claude 3.5 Sonnet on Rust
3. **Requires next-token probability access** — incompatible with commercial black-box APIs
4. TypeScript subset is incomplete (427K lines in full compiler, their coverage is partial)
5. **No discussion of dynamic languages, gradual typing, or applicability beyond static type systems**

### How to cite in the talk
Safe: "In a TypeScript evaluation with open-weight models, 94% of compilation errors were type-check failures — Mundler et al., PLDI 2025"
Don't say: "94% of all AI code errors are type errors"

## Source 3: Endoh 13-Language AI Coding Benchmark (March 2026)

**Author:** Yusuke Endoh (@mame, Ruby committer)
**Repo:** github.com/mame/ai-coding-lang-bench
**Blog:** dev.to/mame/which-programming-language-is-best-for-claude-code-508a

### Methodology
Task: implement "mini-git" (init, add, commit, log, status, diff, checkout, reset, rm, show). Model: Claude Code with Opus 4.6. 20 trials per language per phase (v1 greenfield + v2 extension). 15 configurations × 2 phases × 20 trials = **600 total runs**. 40 shell-based tests per language. No external libraries.

### Type Checker Overhead
- **Ruby/Steep vs Ruby:** 2.55× slower, 2.33× more expensive. v1: 3.2× slower (105s vs 33.2s). Required 20.2 API turns in v1 vs Ruby's 6.0.
- **Python/mypy vs Python:** 1.68× slower, 1.5× more expensive. More stable (±19.0s vs ±69.7s).
- **TypeScript vs JavaScript:** 1.64× slower, 1.59× more expensive. JS ranked #3; TS ranked #11.

### Key Quotes from Endoh
- *"Dynamic languages are consistently faster and more stable."*
- *"Type errors are among the easiest bugs to detect and fix. If an agent frequently introduced type errors without a type checker, it would likely introduce logic bugs at a similar rate."*
- On Steep overhead: *"Ruby/Steep's larger overhead (2.0-3.2×) vs. Python/mypy (1.6-1.7×) may also reflect lower AI familiarity with Steep compared to mypy."*
- *"The classic strategy — start with a dynamic language, then migrate to a static one as the project matures — may still be the right call."*

### Caveats (from the author)
- Greenfield bias — static typing may shine at larger scale
- Small task scope, single model only, no external dependencies
- *"I'm a Ruby committer — take interpretations with a grain of salt"*
- March 2026 snapshot; results change with AI progress

## Source 5: Spiza & Hanenberg, MODULARITY 2014

**Paper:** "Type names without static type checking already improve the usability of APIs (as long as the type names are correct)"
**DOI:** 10.1145/2577080.2577098
**Note:** Paper is behind ACM paywall. Findings from abstract and secondary citations.

- 20 participants
- Measured development time on tasks using unknown/undocumented APIs
- Three conditions: correct type names, wrong type names, no type names
- **Finding 1:** Developers benefit from type names even without static type checking
- **Finding 2:** *"Already a single wrong type name has a measurable significant negative impact on the development time in comparison to APIs without type names"*
- **Implication:** Type annotations in documentation help. But wrong types are worse than no types — quality matters.

## Source 6: Shopify Sorbet Adoption + Boba

### Sorbet Adoption (shopify.engineering/adopting-sorbet, Nov 2020)
- 37,000 Ruby files, 622,000 methods, 2M+ call sites, ~500 gem deps
- 80% of files at `typed: true` or higher
- ~50% of methods have signatures, ~50% of calls typed
- Production NameErrors "dropped to zero" after enabling Sorbet
- 80% of developers want more code to be typed
- **No mentions of AI** — post predates AI coding era (2020)

### Roast/Boba (shopify.engineering/introducing-roast, June 2025)
- Roast: AI workflow framework interleaving deterministic and AI steps
- Boba: specific workflow that auto-adds Sorbet type annotations to test files
- Workflow: sed cleanup → bump to strict → Sorbet autocorrect → AI agent (Claude Code) fixes remaining errors
- Scale: "thousands of test files" — no specific aggregate metrics published
- **This is the AI connection** to Shopify's typing story

## Counter-Evidence Summary

### Training data confound (MultiPL-E, Cassano et al. 2022)
- Removing type annotations from Python: NO significant effect on pass@1 (p=0.23)
- Performance correlates with language popularity, not type system
- Rebuttal: Mundler shows type constraints help at decoding time, independent of training

### Types help compilation, not logic
- -50% compilation errors, but only +3.5-5.5% functional correctness
- +37% for repair tasks (strongest result)
- *"An LLM can generate perfectly typed code with completely wrong logic"*

### RBS adoption is 0.1%
- 170 gems with RBS definitions out of ~175K total
- Most stubs heavily use `untyped`
- Inline RBS is the answer to the separate-file maintenance burden

### Wrong types are harmful
- Spiza & Hanenberg 2014: single wrong type name measurably hurts
- Quality of annotations matters more than quantity

## Strongest Talk-Ready Claims (verified)

1. **"In TypeScript, 94% of LLM compilation errors are type failures"** — Mundler et al., PLDI 2025. Qualifier: TypeScript-only, open-weight models, compilation errors only.

2. **"Ruby is #1 for AI code generation by cost"** — Endoh benchmark, $0.36/run vs $0.62 TypeScript. But Steep adds 2-3× overhead.

3. **"Type names in documentation improve usability even without enforcement — but wrong types are worse than none"** — Spiza & Hanenberg, MODULARITY 2014.

5. **Shopify Boba uses AI to add Sorbet types at scale** — shopify.engineering/introducing-roast, June 2025. Demonstrates the virtuous cycle.
