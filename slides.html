What RDoc does

  • Ruby's default documentation tool — ships with Ruby since 1.8
  • Parses Ruby source, C extensions, and markup files
  • Generates HTML docs and ri terminal docs
  • Supported markup: RDoc, Markdown, RD, TomDoc
  • Powers docs.ruby-lang.org
Quick primer. Most people have seen RDoc output but don't think about the tool itself.

Some context

RDoc has been Ruby's documentation tool for 20+ years. It works. But a lot has changed since it was written.

  • Developers expect dark mode, mobile support, live preview
  • Markdown became the standard way to write documentation
  • Ruby syntax grew beyond what the parser was built for
  • Type annotations exist now — documentation doesn't show them
Be factual about where things stood. Not blaming — the ecosystem changed.

Why Markdown?

  • Markdown is the industry standard
  • RDoc markup is Ruby-specific — even within the community, it's not widely known
  • Many contributors already write Markdown in RDoc comments without realizing it doesn't render correctly

If you can write a GitHub README, you can contribute to Ruby documentation.

Markdown was added ~2012 by reusing RDoc's pipeline. Every Markdown fix is a two-format fix.

Two parsers feeding into one shared pipeline. Markdown parser outputs RDoc markup strings. Shared InlineParser re-parses them.

Example: ~~strikethrough~~

# Markdown parser outputs:
~~text~~  →  "<del>text</del>"  # plain string!

# Shared InlineParser receives that string
# ...but didn't recognize <del>
# Strikethrough was silently broken for years

# Fix: modify the shared InlineParser
#   → must verify RDoc markup still works too

Every Markdown fix is a two-format fix.

One concrete example. Fix required changing shared InlineParser — must test both formats.

What's improved

  • ~~Strikethrough~~ aligned with GFM spec
  • GitHub-style heading anchors (#heading-name)
  • Table parsing: inline markdown, incomplete rows
  • Bash/shell syntax highlighting
  • Backtick quoting in RDoc markup too
  • break_on_newline enabled by default

RDoc markup vs Markdown

RDoc markup

= Heading
*bold*  _italic_  +code+
{link text}[url]
* bullet item

Markdown

# Heading
**bold**  *italic*  `code`
[link text](url)
- bullet item
Same documentation in both formats. Markdown immediately recognizable to any developer.

Migration plan

1Get Markdown to GFM-level quality ← in progress
2Build migration tooling (RDoc markup → Markdown)
3Migrate ruby-core documentation
4Deprecate RDoc markup? (open question)

Can RBS replace :call-seq:?

:call-seq:

# :call-seq:
#   readlines(sep=$/)     -> array
#   readlines(limit)      -> array
#   readlines(sep, limit) -> array

RBS #:

#: (?String sep) -> Array[String]
#: (Integer limit) -> Array[String]
#: (String, Integer) -> Array[String]

The overlap is significant. For a language that doesn't want typing, writing type signatures for documentation is a big shift. But I think it's the right direction.

Raise as a question, not statement. For a language that doesn't want typing, this is significant.

The gap: default values

:call-seq:

commercial(cwyear, cweek=1, cwday=1, sg=nil)

Shows actual default values

RBS:

(Integer, ?Integer, ?Integer, ?Symbol?) -> Date

Only says "optional" — default is lost

If we want to migrate, RBS needs default value support.

Concrete blocker. call-seq captures actual default values. RBS can't express this.

Where this stands

  • I'm aiming to ship RBS display on docs.ruby-lang.org
  • The call-seq migration depends on RBS supporting default values — that's not solved yet
  • This needs community discussion, not just my opinion

But the direction seems clear: structured, machine-readable signatures are better than free-form text.

Status update. call-seq migration needs community input.
03

Getting ahead

How do we avoid falling behind again?

llms.txt: I prototyped it

Then I looked at the data:

  • 10% adoption across 300K domains
  • Zero measurable impact on AI citations
  • ML model improved when llms.txt was removed
  • Zero bot visits from GPTBot, ClaudeBot, etc.
  • No major LLM provider officially supports it

The trade-off is generation time and code complexity for every RDoc user, for a benefit that doesn't exist yet. So I decided not to pursue it — at least for now.

Sources: SE Ranking 300K domain study. XGBoost improved when llms.txt removed. Zero bot visits.

What I think actually helps

Instead of generating special files for LLMs, I'd rather make the documentation itself better:

  • Markdown — any tool can parse it, humans and machines alike
  • RBS types — structured data instead of free-form text
  • Clean architecture — makes it easier to build things we haven't thought of yet

On the tooling side, I've been building github.com/st0012/ruby-skills — Claude Code plugins for Ruby development. No RDoc-specific skills yet, but that's coming once the Markdown work is further along.

Don't chase trends. Build foundations for all consumers. ruby-skills has 109 stars.

How I think about this

  • Check the data before building a feature — llms.txt looked promising until I looked at the numbers
  • Invest in foundations that serve multiple purposes — Markdown, types, and clean architecture all make docs better for everyone, including future tools
  • As a maintainer, I'd rather ship fewer things that are solid than chase every trend
Tie it together. Data over hype. Foundations over band-aids.
